## Vue d'ensemble

Payfonte fournit des API pour trois capacites principales :

- **Collectes** : creer des checkouts de paiement, lancer des direct charges et verifier les transactions.
- **Decaissements** : valider des beneficiaires, demander des virements, lister les decaissements et verifier leur statut.
- **Portefeuilles** : recuperer les soldes de portefeuille par `country` et `currency`.

La specification API definit **11 operations** sur les flux de collecte, de decaissement et de portefeuille.

## URL de base

Utilisez le bon serveur selon votre environnement :

| Environnement | URL de base |
| --- | --- |
| Sandbox | `https://sandbox-api.payfonte.com` |
| Production | `https://api.payfonte.com` |

## Authentification

Tous les endpoints exigent deux en-tetes de cle API, et non une authentification Bearer :

- `client-id`
- `client-secret`

```bash
curl --request GET 'https://sandbox-api.payfonte.com/payments/v1/checkouts' \
  --header 'client-id: <your-client-id>' \
  --header 'client-secret: <your-client-secret>'
```

## Cartographie des endpoints

### Collectes

| Methode | Chemin | Objet |
| --- | --- | --- |
| `POST` | `/payments/v1/checkouts` | Generer une URL de checkout |
| `GET` | `/payments/v1/checkouts` | Lister les transactions de checkout (paginees) |
| `POST` | `/payments/v1/payments/direct-charge` | Lancer un direct charge via un provider |
| `GET` | `/payments/v1/payments/direct-charge/{provider}/properties` | Recuperer les proprietes specifiques au provider |
| `GET` | `/payments/v1/payments/verify/{reference}` | Verifier une transaction de collecte |

### Decaissements

| Methode | Chemin | Objet |
| --- | --- | --- |
| `GET` | `/billing/v1/transfer-recipients/{provider}/properties` | Recuperer les proprietes du beneficiaire/provider |
| `POST` | `/billing/v1/transfer-recipients/validate` | Valider un beneficiaire de virement |
| `POST` | `/billing/v1/disbursements` | Demander un decaissement |
| `GET` | `/billing/v1/disbursements` | Lister les transactions de decaissement (paginees) |
| `GET` | `/billing/v1/disbursements/verify/{reference}` | Verifier le statut d'un decaissement |

### Portefeuilles

| Methode | Chemin | Objet |
| --- | --- | --- |
| `GET` | `/billing/v1/wallets` | Recuperer les soldes des portefeuilles |

## Conventions communes des requetes

- **Format des montants** : envoyez `amount` dans la plus petite denomination (sous-unite).
  Exemple : `100.00` USD s'envoie sous la forme `10000`.
- **Devise** : code sur 3 caracteres (par exemple `NGN`, `USD`, `XOF`, `XAF`).
- **Pays** : code sur 2 caracteres (par exemple `NG`, `US`, `CI`, `SN`).
- **Reference** : la reference generee par le marchand doit etre unique par requete.
- **Surcharge webhook** : certains endpoints acceptent une URL de webhook specifique a la requete.

## Flux habituels

### Flux de collecte

1. Creez un checkout (`POST /payments/v1/checkouts`) ou lancez un direct charge (`POST /payments/v1/payments/direct-charge`).
2. Recevez les mises a jour via webhook et/ou interrogez la verification (`GET /payments/v1/payments/verify/{reference}`).
3. Utilisez l'endpoint de liste (`GET /payments/v1/checkouts`) pour le rapprochement et le reporting.

### Flux de decaissement

1. Recuperez les proprietes du provider si necessaire puis validez le beneficiaire (`POST /billing/v1/transfer-recipients/validate`).
2. Demandez le decaissement (`POST /billing/v1/disbursements`).
3. Verifiez le statut (`GET /billing/v1/disbursements/verify/{reference}`) et rapprochez avec l'endpoint de liste (`GET /billing/v1/disbursements`).

## Pagination et filtres

Les endpoints de liste prennent en charge la pagination avec `page` et `limit`.
Les endpoints de liste de collecte et de decaissement prennent aussi en charge les filtres de date (`dateFrom`, `dateTo`).
